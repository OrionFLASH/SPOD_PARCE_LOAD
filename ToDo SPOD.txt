П.1 — Проверка SPOD-JSON по листу/колонке
  ГДЕ В КОНФИГЕ: файл config.json → объект "consistency_checks" → массив "rules"
  → шесть правил с "type": "json_spod_format" и id spod_json_contest_data,
  spod_json_contest_period, spod_json_reward_add_data,
  spod_json_tournament_filter_period, spod_json_tournament_target_type,
  spod_json_indicator_filter (сразу после csv_columns_count). Подсказка в том же
  разделе: ключ "spod_todo_config_guide".
  СДЕЛАНО: тип правила json_spod_format (consistency_checks.py), модуль
  src/json_spod_format_check.py, правила в config.json (id spod_json_*).
  Как: снятие внешних кавычек у всей ячейки, замена """ -> ", контроль
  пустого значения после ключа, список numeric_value_keys (число без """),
  разбор JSON; эвристика ключа в одной паре " вместо тройных кавычек.
  ОГОВОРКИ: нет полного обхода всего текста с требованием «каждый ключ и
  каждая строковая величина строго в """…""" на всех уровнях вложенности» —
  часть нарушений отлавливается только если ломается JSON или попадает под
  эвристики. Смешанная нотация (как в примере с двойными "" у ключей в
  FILTER_PERIOD_ARR) может остаться незамеченной, если после нормализации JSON
  всё же валиден. Массивы: требование «элементы в """» не проверяется
  отдельным проходом, опираемся на итоговый JSON.

П.2 — Ограничение строк для referential / referential_composite
  ГДЕ В КОНФИГЕ: тот же массив consistency_checks.rules — у боевых referential
  добавлены пустые "src_row_conditions": [], "ref_row_conditions": [] (см. правила
  id ref_group_contest_code_in_contest_data, ref_indicator_contest_code_in_contest_data,
  ref_reward_link_contest_code_in_contest_data, ref_reward_link_reward_code_in_reward,
  ref_employee_org_unit_code_in_org_unit_v20); заполните при необходимости. Полный образец условий
  из текста задачи — правило id example_referential_row_filters (enabled: false),
  скопируйте блоки условий в своё правило и включите его.
  СДЕЛАНО: src_row_conditions и ref_row_conditions (алиасы
  sheet_src_row_conditions, sheet_ref_row_conditions): объекты
  { "column", "op", "value" }, op = или <> (также ==, !=, eq, ne). Все
  условия — логическое И. Строки источника вне фильтра: в колонке результата
  «—», не входят в violations. Справочник строится по отфильтрованным
  строкам ref.
  ОГОВОРКИ: нет одного поля с массивом «любое из значений» — несколько
  вариантов задаются несколькими элементами в массиве условий (или расширять
  конфиг позже). Пример из текста ниже (BUSINESS_STATUS + CONTEST_TYPE +
  CREATE_DT) в config.json не добавлялся — только механизм.

П.3 — CONSISTENCY при enabled: false
  СДЕЛАНО: строка в своде сохраняется, total_rows по целевому листу,
  violations = 0, в sample текст на русском про отключённое правило
  (не «disable», а явная формулировка; флаг disabled в записи результата).

П.4 — Консоль и лог: архив SQLite
  СДЕЛАНО: путь к БД — относительный от корня проекта, полное имя файла,
  перенос по ширине терминала (без «…» в середине пути). Блок «По листам» —
  заголовок-таблица (Лист / Строки / Примечание). Стартовая строка лога
  архива использует тот же относительный путь.
  ОГОВОРКИ: при узком терминале длинные подписи листов в таблице могут
  усекаться; режим verbose по-прежнему может обрезать вторичные строки
  через общий лимит ширины.

П.5 — RATING, ignoreConditions
  СДЕЛАНО: разбор ignoreConditions в reward_item_catalog (каталог ITEM),
  item_accessible_for_manager(..., manager_tab=), rating_item_matrix передаёт
  табельный строки; при совпадении ячейка считается доступной (заливка
  fill_accessibility_ok).

те надо проверять все ли параметры с экранированием, если какой-то из параметров не экранирован с обеих сторон или значение не экранировано
то это считаем за ошибку консистентности

также вся строка может быть заключена в одинарную кавычку - это верно
не верно если кавычки не с обеих сторон

также и экранирование - тоже должно быть с двух сторон

те надо проверить наличие тройных кавычек, наличие (если есть) начальной и конечной кавычки

после надо сделать в памяти замену """ -> "  тройных кавычек на одинарные и проверить сам JSON на валидность.
если валиден то ОК если нет, то это тоже ошибка консистентности

вот так верно ("""momentRewarding""": """""", )
а так нет ("""momentRewarding""": , )

также в параметрах проверки на консистентность надо задать параметры (перечислить их), значения которых не должны иметь экранирования вообще
это параметры значения которых должны быть числовыми
те для них (те что укажем) название параметра все равно в экране. А вот значение без
те для них вот так - верно: """period_code""": 1,
а вот так - нет: """period_code""": """1""",
