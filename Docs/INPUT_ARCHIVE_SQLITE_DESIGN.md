# Архив входных CSV в SQLite (`input_archive_sqlite`)

> **Построчный режим (v2):** при **`row_level_archive`: true** используется отдельный файл БД и модуль **`src/input_archive_sqlite_v2.py`**. Подробная документация реализации — **`Docs/INPUT_ARCHIVE_ROW_LEVEL.md`**. Ниже в основном описан режим **v1** (снимки целого файла).

## 1. Возможность и выбор СУБД

**Да, основную программу можно дополнить записью загружаемых файлов в базу без отдельного сервера.**

- **SQLite** хранится в одном файле; в Python используется модуль **`sqlite3`** из стандартной библиотеки — **дополнительные пакеты не нужны** (в отличие от PostgreSQL/MySQL).
- Точка интеграции естественная: сразу после параллельного чтения CSV в **`main_impl.main()`**, когда для каждого листа уже есть **сырой** `DataFrame` (как после `read_csv_file`, **до** разворота JSON) и путь к файлу на диске.

«Little SQL» в смысле лёгкой СУБД для этой задачи — это и есть **SQLite**.

## 2. Логика версий и отсутствие дублей при каждом запуске

- Для каждой тройки **лист (`sheet`) + имя файла из конфига + `subdir`** в **`archive_file_inventory`** хранится **сводка по последнему известному файлу**: размер, mtime, число строк/колонок, **SHA-256 содержимого** (байты CSV на диске), id актуального снимка, даты проверки/загрузки, счётчики **сколько раз пропустили запись** (`total_skips_same_content`) и **сколько раз делали ingest** (`total_ingests`).
- В **`archive_file_snapshot`** по-прежнему не более одной строки **`latest`** на эту тройку; исторические снимки — **`historical`**.

**Когда новый снимок (новые строки в `arch_*`) не создаётся:**

- **`use_sha256_for_identity`: true** (рекомендуется): если **размер**, **число строк** и **число колонок** совпадают с `archive_file_inventory`, считается **SHA-256 файла** и сравнивается с `last_content_sha256`. Совпадение → обновляются только **`last_checked_at`**, пути, mtime в сводке и **`actuality_checked_at`** у снимка; **таблица данных не растёт**. Так отсекаются повторные запуски при неизменных CSV (в т.ч. если менялся только код программы).
- Если размер/строки/колонки **уже** отличаются от inventory → новый снимок **без предварительного хеша** (экономия чтения файла).
- **`use_sha256_for_identity`: false**: сравнение только **размер + строки + колонки + mtime** (как раньше); смена mtime без смены содержимого может дать лишний снимок.

**Первый запуск после обновления кода:** для старых БД без SHA в inventory при совпадении размеров/строк/колонок хеш **досчитывается и записывается** в inventory и в `archive_file_snapshot` **без** повторной вставки строк данных — **но только если** у текущего latest-снимка в `source_sha256` ещё `NULL` **или** он совпадает с только что посчитанным хешем файла. Иначе (файл на диске уже другой при тех же размерах) выполняется полный ingest.

**Согласованность inventory и latest:** пропуск ingest по SHA допускается, если совпадают и `last_content_sha256` в inventory, и `source_sha256` у latest (либо в снимке SHA ещё не было). При расхождении — либо предупреждение и новый снимок, либо синхронизация inventory без ingest, если файл совпадает с latest.

- Если отличается содержимое (или метаданные в режиме без SHA) — прежний `latest` → **`historical`**, новый снимок и строки в `arch_*`.

Параметр **`append_on_content_change`**: при `false` в базу попадает **только первый** снимок для данного файла/листа.

## 3. Структура данных

### Мета-таблица `archive_file_snapshot`

| Поле | Назначение |
|------|------------|
| `id` | Первичный ключ снимка |
| `sheet_name`, `file_name`, `subdir` | Логическая идентичность входа из `input_files` |
| `resolved_path` | Полный путь к CSV при загрузке |
| `source_mtime`, `source_size`, `source_row_count` | Сигнатура для сравнения |
| `source_col_count` | Число колонок CSV (после чтения) |
| `source_sha256` | SHA-256 байтов файла (при включённом режиме идентичности) |

### Таблица `archive_file_inventory` (сводка по логическому файлу)

Одна строка на `(sheet_name, file_name, subdir)`: последняя сигнатура, `latest_snapshot_id`, `last_ingest_at`, `last_checked_at`, счётчики прогонов. По ней удобно смотреть, **обновлялась ли БД** и как часто срабатывал пропуск как «тот же файл».
| `loaded_at` | UTC ISO8601 — когда строки записаны в БД |
| `actuality_checked_at` | Когда последний раз сравнивали с диском |
| `row_status` | `latest` или `historical` |
| `superseded_by_id` | Для `historical`: id нового снимка, заменившего этот |

### Таблицы по листам `arch_<SHEET>` (например `arch_CONTEST_DATA`)

- Системные колонки (имена задаются в `system_columns`): **`__snapshot_id`**, **`__row_ix`**, **`__loaded_at`**.
- Остальные колонки — **TEXT**, имена приведены к безопасным для SQL идентификаторам; значения — как строки из CSV (как в пайплайне после чтения).
- При появлении **новых** колонок в CSV выполняется **`ALTER TABLE ADD COLUMN`**.

#### Разворот JSON для **CONTEST-DATA** и **REWARD** (дополнительные колонки в конец таблицы)

Для листов **`CONTEST-DATA`** и **`REWARD`** в конец строки (после всех колонок из CSV) добавляются поля с префиксом **`JSON_`**, построенные из колонок **`CONTEST_FEATURE`** и **`REWARD_ADD_DATA`** соответственно (см. **`Docs/JSON/SPOD_INPUT_DATA_CATALOG.md`**: предобработка тройных кавычек, деревья путей).

- **Исходная** колонка CSV (**`CONTEST_FEATURE`** / **`REWARD_ADD_DATA`**) **не удаляется** — в таблице остаётся исходная строка ячейки и рядом развёрнутые значения.
- **Нормализация** перед `json.loads`: замена **`"""` → `"`**, снятие внешних кавычек, если вся ячейка — JSON-объект/массив в кавычках; далее **`safe_json_loads`** из **`src/json_utils.py`** (типовые исправления строки).
- **Структура имён**: только **листья** дерева — скаляры; вложенные объекты обходятся по ключам; массив **только примитивов** даёт одно поле со значениями через **`"; "`**; массив объектов — индексы **`0`**, **`1`**, … в пути. Имя колонки: **`JSON_<имя_исходной_колонки>_<ключ1>_<ключ2>_…`** (сегменты пути санируются под SQLite). При коллизии с уже занятым именем из CSV добавляется суффикс **`_j2`**, **`_j3`**, ….
- **Появление колонок**: недостающие **`JSON_*`** добавляются через **`ALTER TABLE ADD COLUMN`** не только при новом снимке, но и при **каждом** прогоне архива для данного листа (после успешного чтения `DataFrame`), чтобы схема догоняла текущий код даже при **пропуске ingest** («тот же файл» по SHA).
- **Заполнение значений**: при **новом** снимке значения **`JSON_*`** пишутся вместе с **`INSERT`**. Если новый снимок **не** создаётся (без изменений содержимого, дозапись SHA в inventory, синхронизация метаданных, реактивация **`reuse_matching_historical_snapshot`**), для строк **актуального** снимка выполняется **`UPDATE arch_* SET JSON_… = ? WHERE __snapshot_id = ? AND __row_ix = ?`** по данным текущего чтения CSV — тогда **`JSON_*`** не остаются **NULL** из‑за того, что разворот JSON появился в коде позже первой загрузки.

Реализация: **`src/archive_json_columns.py`** (**`plan_archive_json_flat_columns`**, **`update_json_flat_for_snapshot_rows`**); **`src/input_archive_sqlite.py`** вызывает разворот при вставке снимка и синхронизирует **`JSON_*`** перед **`commit`** на ветках без полного re-ingest.

## 4. Связи между таблицами

По требованию **не вводить FK**, если связь неочевидна: **внешние ключи между таблицами листов не создаются**. Связи **CONTEST_CODE**, **REWARD_CODE**, **TOURNAMENT_CODE** и т.д. остаются **обычными полями** в данных; выборки «как JOIN в Excel» делаются запросами при анализе.

При необходимости позже можно добавить **представления (VIEW)** или отдельный слой отчётов — без изменения схемы хранения.

## 5. Конфигурация `config.json`

Секция **`input_archive_sqlite`** (см. актуальный `config.json`):

- **`row_level_archive`** — при **`true`** активен **v2** (`input_archive_sqlite_v2.py`, **`db_path`** → обычно **`OUT/DB/spod_input_archive_v2.sqlite`**). При **`false`** — **v1** (снимки), путь БД берётся из **`legacy_db_path`** (**`OUT/DB/spod_input_archive.sqlite`**).
- **`enabled`** — мастер-переключатель.
- **`db_path`** — путь к файлу БД относительно корня проекта или абсолютный (для v1 по умолчанию **`OUT/DB/spod_input_archive.sqlite`**; для v2 в поставке — **`spod_input_archive_v2.sqlite`**).
- **`legacy_db_path`** — файл БД v1, если построчный режим выключен.
- **`default_row_key_by_sheet`**, **`parallel_row_processing`**, **`skip_ingest_if_file_unchanged`**, **`row_hash_columns`** — только для v2; см. **`Docs/INPUT_ARCHIVE_ROW_LEVEL.md`**.
- **`default_archive_to_db`** — значение по умолчанию, если у записи в **`input_files`** **нет** ключа **`archive_to_db`**. В поставляемом **`config.json`** стоит **`false`**, а у **каждой** записи **`input_files`** явно указано **`archive_to_db`**: так проще видеть политику по файлу. Если поставить **`true`**, в архив пойдут все записи без своего ключа; для исключения отдельного файла задайте **`"archive_to_db": false`**.
- У каждого элемента **`input_files`** можно задать:
  - **`archive_to_db`**: `true` / `false` — грузить ли этот файл в архив.
  - **`archive_db_path`**: относительный (от корня проекта) или абсолютный путь к **другому** файлу SQLite для этого файла. Если ключ отсутствует или пустой — используется **`input_archive_sqlite.db_path`**. Разные пути в одном прогоне дают **несколько** последовательных записей (по одной БД на группу файлов); схема и логика снимков в каждой БД те же.
- **`append_on_content_change`**: при `false` — только первый снимок на файл.
- **`use_sha256_for_identity`**: сравнение «тот же файл» по SHA-256 **байтов CSV** при совпадении размера/строк/колонок (по умолчанию **true**).
- **`reuse_matching_historical_snapshot`** (по умолчанию **true**, имеет смысл при включённом SHA): перед созданием нового снимка ищется **любой** прошлый снимок с тем же **`source_sha256`**. Если найден **не** текущий `latest`, он снова помечается как **`latest`**, прежний `latest` уходит в **`historical`** (с `superseded_by_id` на «возвращённый» снимок), **без** `INSERT` строк в `arch_*` — исключает дубли при откате файла к старому содержимому. Если совпадает текущий `latest`, выполняется только синхронизация метаданных (как при «без изменений»). Без SHA-256 этот механизм **не** используется (по метаданным нельзя безопасно отличить «тот же CSV» после смены mtime).
- **`compute_sha256`**: устаревший алиас; если задан только он и нет `use_sha256_for_identity`, подставляется в режим идентичности.
- **`system_columns`**: переимена системных полей в таблицах данных.
- **`reporting`** — независимые режимы подробности **консоли** и **лог-файла**:
  - **`console`**: `off` — не печатать блок архива в stdout; `summary` — только заголовок и **итоговые счётчики** (новый снимок / без изменений / дозапись SHA / пропуски / ошибки / вне архива / нет данных); `normal` — сводка **плюс строка на каждый лист** (результат, число строк); `verbose` — как `normal`, плюс в консоли размер файла, id снимка, префикс SHA, пояснения.
  - **`log`**: `minimal` — **построчные** сообщения о решениях только уровнем **DEBUG** (в логе при `logging.level=INFO` почти не видны), плюс одна строка **INFO** с **итогом** прогона; `normal` — каждое событие **INFO**, детали (путь, полный SHA, размеры) — **DEBUG**; `verbose` — **INFO** + расширенный **DEBUG** (в т.ч. путь к файлу, полный хеш).

Файл БД лежит под **`OUT/DB/`**; каталог **`OUT/`** целиком в **`.gitignore`** — локальная БД в репозиторий не попадает. Старый путь **`archive/`** в **`.gitignore`** оставлен на случай локальных копий.

## 6. Примеры запросов

**Последняя реплика листа CONTEST-DATA:**

```sql
SELECT s.id, s.loaded_at, s.source_row_count
FROM archive_file_snapshot s
WHERE s.sheet_name = 'CONTEST-DATA' AND s.row_status = 'latest';
```

**Строки только актуального снимка:**

```sql
SELECT d.*
FROM arch_CONTEST_DATA d
JOIN archive_file_snapshot s ON s.id = d.__snapshot_id
WHERE s.sheet_name = 'CONTEST-DATA' AND s.row_status = 'latest';
```

**Все снимки за период по времени загрузки:**

```sql
SELECT * FROM archive_file_snapshot
WHERE loaded_at >= '2026-03-01T00:00:00Z' AND loaded_at < '2026-04-01T00:00:00Z'
ORDER BY loaded_at;
```

## 7. Реализация в коде

- **`src/input_archive_sqlite.py`** — создание схемы, сравнение сигнатур, вставка; сообщения в лог по **`reporting.log`**, сбор строк для консоли. В лог и в отчёт передаётся **`db_display`**: полный путь к файлу БД **относительно корня проекта** (без усечения «…» в середине пути).
- **`src/archive_json_columns.py`** — нормализация и разворот **`CONTEST_FEATURE`** / **`REWARD_ADD_DATA`** в колонки **`JSON_*`** в конец `arch_*`; при необходимости — **`UPDATE`** существующих строк снимка.
- **`src/console_ui.py`** — **`print_input_archive_sqlite_report`**: вывод в stdout по **`reporting.console`** (независимо от уровня логгера). Режимы: **`off`** — тишина; **`summary`** — заголовок, **полный** путь к БД (перенос по ширине терминала), итоговые счётчики; **`normal`** — плюс **таблица** «Лист / Строки / Примечание» по событиям архива; **`verbose`** — плюс размер файла, префикс SHA, id снимка.
- **`src/main_impl.py`** — после фазы чтения CSV вызов архива; `process_single_file` возвращает ещё **`file_path`** для метаданных диска.
- **`src/config_loader.py`** — атрибут **`input_archive_sqlite`** (смёрженный с дефолтами).

## 8. Ограничения и замечания

- Запись идёт **последовательно** в одном соединении (после параллельного чтения файлов), режим **WAL** снижает блокировки.
- Очень большие CSV увеличивают файл `.sqlite`; при необходимости — ротация файла БД по пути или выгрузка старых снимков.
- Имена колонок с спецсимволами приводятся к безопасным SQL-именам; исходное имя в метаданных таблицы не дублируется — ориентир по порядку и документации выгрузки.

## 10. Режим v2 (построчный архив) — краткая сводка

| Тема | v1 | v2 |
|------|----|----|
| Модуль | `input_archive_sqlite.py` | `input_archive_sqlite_v2.py` |
| Гранулярность | Снимок всего файла | Строка по `row_key_hash` |
| Таблицы данных | `arch_<SHEET>` + `__snapshot_id` | `archive_row_current`, `archive_row_payload` |
| Пропуск без изменений | SHA **файла** | SHA **файла** (inventory) + сравнение **хешей строк** |
| Отчёт в консоли | `print_input_archive_sqlite_report` | `print_input_archive_row_report` |

Полное описание: **`Docs/INPUT_ARCHIVE_ROW_LEVEL.md`**.
