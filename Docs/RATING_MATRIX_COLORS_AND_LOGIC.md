# Лист RATING: подсчёт значений, доступность и раскраска (подробно)

Документ описывает **фактическое** поведение кода на дату реализации: `src/rating_item_matrix.py`, `src/reward_item_catalog.py`, секция **`rating_item_matrix`** в `config.json`, этапы в **`src/main_impl.py`**.

Связанные материалы: `Docs/RATING_MATRIX_ENRICHMENT_PLAN.md` (план), `README.md` (кратко).

---

## 1. Что вообще раскрашивается на листе RATING

На листе **RATING** (объединённый из файлов `RATING_*` через `aggregate_into_sheet`) есть **два независимых** механизма оформления.

| Механизм | Что красит | Где в config |
|----------|------------|--------------|
| **`color_scheme`** | Только **заголовки** (строка 1) перечисленных колонок | `color_scheme[]` |
| **`column_formats`** | Не цвет: выравнивание, формат числа/даты | `column_formats[]` |
| **`rating_item_matrix`** | **Фон ячеек** в добавленных колонках товаров ITEM + **фон заголовка** этих колонок | `rating_item_matrix` |

**Важно:** лист **RATING** **не входит** в группы `color_scheme` с заливкой «Исходные данные» и т.п. (там только PROM-листы вроде CONTEST-DATA, REWARD). Исходные колонки рейтинга (табельный, роль, период, места, кристаллы) **без фоновой заливки** — только форматирование из `column_formats` для листа `"sheet": "RATING"` (числа по центру, разделитель тысяч).

**Цветные ячейки данных** на RATING — это **только** колонки матрицы ITEM, добавленные шагом `apply_rating_item_matrix_enrichment`.

## 2. Порядок выполнения в `main.py`

```text
1. Чтение CSV → merge в лист RATING (aggregate_into_sheet).
2. merge_fields_advanced, сводка REWARD (getCondition) — без раскраски RATING.
3. apply_rating_item_matrix_enrichment(sheets_data)
   → в DataFrame RATING добавляются колонки ITEM;
   → в памяти строится meta: matrix_cells, header_stock_out_columns, fills.
4. write_to_excel → для каждого листа apply_color_scheme (на RATING почти ничего не красит).
5. apply_rating_item_matrix_colors(output_excel, meta)
   → openpyxl перезаписывает fill у ячеек матрицы и заголовков ITEM-колонок.
```

Подсветка матрицы **всегда последняя** для ITEM-колонок — она не затирается `color_scheme`.

## 3. Откуда берутся колонки матрицы ITEM

### 3.1. Источник списка колонок

1. Лист **REWARD**, строки с `REWARD_TYPE` = **ITEM** (регистр не важен).
2. Для каждой строки: `REWARD_CODE`, опционально сезон из развёрнутой колонки  
   `ADD_DATA => getCondition => employeeRating => seasonCode`.
3. Имя колонки на RATING: `REWARD_CODE` или `REWARD_CODE (seasonCode)`; при дубликатах — суффикс `__2`, `__3`.

Колонки **сортируются** по коду и сезону. Порядок в Excel = этот порядок (в конец таблицы RATING).

### 3.2. Пороги рейтинга для колонки

Из той же строки REWARD (или из JSON `REWARD_ADD_DATA` через каталог):

| Поле в meta / REWARD | Смысл |
|----------------------|--------|
| `min_bank` | Макс. допустимое **место в рейтинге по стране** (чем меньше число, тем лучше) |
| `min_tb` | Макс. место **ТБ** |
| `min_gosb` | Макс. место **ГОСБ** |

Если колонка на RATING **не найдена**, соответствующий порог **не проверяется** (как «нет ограничения»). Если колонка есть, но ячейка пуста/не число — порог **считается не выполненным** (доступность ↓).

## 4. Подсчёт заказов (число в ячейке)

### 4.1. Лист ORDER

- Берётся объединённый лист **ORDER** (`sheet_order` в config, по умолчанию `"ORDER"`).
- Сопоставление столбцов (с запасными именами):
  - **Табельный:** `order_employee_col` → по умолчанию «Табельный номер».
  - **Код товара:** `order_product_col` → «Код товара» / `REWARD_CODE` и др.

### 4.2. Фильтр статуса

Строки ORDER **исключаются**, если `Статус заказа` ∈ `order_status_exclude` (по умолчанию **Отклонён**, **Отменён**).

Если колонка статуса **не найдена** — в лог **WARNING**, в расчёт попадают **все** строки ORDER.

### 4.3. Формула count для ячейки (менеджер M, товар T)

```text
count = число строк ORDER после фильтра, где
        табельный = M  AND  код товара = T
```

Одна строка ORDER = один заказ (+1). Это **не** сумма количества из поля «кол-во», а **число транзакций**.

### 4.4. Признак «заказан»

```text
ordered = (count > 0)
```

## 5. Базовая доступность (`item_accessible_for_manager`)

Функция: **`src/reward_item_catalog.py` → `item_accessible_for_manager`**.

Вход для **каждой строки RATING** (менеджер) и **каждой колонки ITEM** (код `match_code`):

| Вход | Откуда |
|------|--------|
| `rank_country`, `rank_tb`, `rank_gosb` | Колонки места на строке RATING |
| `crystals` | «Количество кристаллов» |
| `order_product_codes` | Множество **всех** кодов товаров из ORDER этого табельного (любой статус после фильтра) |
| `list_reward_codes` | Коды наград с листа **LIST-REWARDS** для этого табельного |
| `manager_tab` | Табельный номер строки RATING |
| `rules` | Каталог ITEM по `REWARD_CODE` + пороги из строки REWARD |

### 5.1. Правила (все должны выполниться, кроме ignoreConditions)

| # | Условие | Логика |
|---|---------|--------|
| 0 | **ignoreConditions** | Если табельный в списке из `REWARD_ADD_DATA` → сразу **доступен** (True), остальное не проверяется |
| 1 | **minRatingBANK** | Если порог > 0: `rank_country` не пуст и **≤** порога |
| 2 | **minRatingTB** | Аналогично для ТБ |
| 3 | **minRatingGOSB** | Аналогично для ГОСБ |
| 4 | **minCrystalEarnedTotal** | Если > 0: кристаллы **≥** порога |
| 5 | **rewardCode** (в getCondition) | Если список не пуст: **каждый** код должен быть в LIST-REWARDS менеджера |
| 6 | **nonRewardCode** | Если список не пуст: **ни один** код не должен быть среди заказанных кодов ORDER |

Порог рейтинга **0** или пусто = проверка **отключена** для этой метрики.

Результат: **`base_accessible`** (True/False).

## 6. Дополнительные ограничения на доступность

### 6.1. Лимит заказов по группам сезона (`item_order_groups`)

Для **текущей строки** менеджера:

1. Считается `count_by_code` — словарь {код товара → число заказов}.
2. Для каждой группы из config (например `SEASON_m_2025_2`, max 2):
   - `total = sum(count_by_code[c] for c in codes группы)`
   - если `total >= max_orders` → **все** коды группы попадают в `blocked_codes`.

```text
accessible_after_groups = base_accessible AND (match_code NOT IN blocked_codes)
```

Пример: менеджер заказал 2 товара из `ITEM_02_2025-2_*` → вся группа SEASON_m_2025_2 блокируется для него, даже если рейтинг «зелёный».

### 6.2. Лимит **itemAmount** (склад товара)

В JSON товара (`itemAmount` в `REWARD_ADD_DATA`) и в тексте feature («Доступно всего 5…») задаётся **число слотов на весь товар**, а не «5 заказов одним человеком».

Режим в config: **`item_amount_scope`** (по умолчанию **`global`**).

#### Режим `global` (по умолчанию)

```text
global_count = сумма заказов по коду товара по ВСЕМ табельным в ORDER (после фильтра статуса)
склад_исчерпан = (itemAmount задан) AND (global_count >= itemAmount)
```

| Ситуация менеджера | accessible | Значение | Шапка колонки |
|--------------------|------------|----------|---------------|
| склад исчерпан, count = 0 | нет | **N** | красная, если склад_исчерпан |
| склад исчерпан, count > 0 | да* | **count** | красная |
| склад не исчерпан | по §5–6.1 | Y/N/count | обычная |

\* при выполнении рейтинга и прочих критериев.

**Пример `ITEM_01_2025-2_08`, itemAmount = 5:** в ORDER пять строк от **пяти разных** табельных (по 1 заказу). Раньше код требовал `count >= 5` у **одного** табельного → шапка не красилась и оставались **Y**. В режиме `global` сумма 5 ≥ 5 → шапка красная, у менеджеров без заказа — **N**.

#### Режим `per_manager`

```text
item_amount_scope: "per_manager"
личный_лимит = count(этот табельный, этот код) >= itemAmount
```

Красная шапка, если **хотя бы одна** строка листа исчерпала личный лимит. Для «Доступно всего N» на витрину обычно нужен **`global`**.

## 8. Четыре цвета тела ячейки (матрица)

Ключ заливки: **`_matrix_fill_key(accessible, ordered)`**.

| accessible | ordered | Ключ config | Смысл | Hex по умолчанию (пастель) |
|------------|---------|-------------|-------|----------------------------|
| да | да | `fill_ordered_available` | Заказывал, критерии OK | `E2F0D9` |
| нет | да | `fill_ordered_unavailable` | Заказывал, но не должен был (рейтинг/группа/лимит) | `F5D6DC` |
| да | нет | `fill_available_not_ordered` | Может заказать, ещё не заказывал | `D9EAD3` |
| нет | нет | `fill_unavailable_not_ordered` | Недоступен, не заказывал | `F2D7DC` |

Цвета задаются в **`rating_item_matrix`** в `config.json` (без `#`, формат RRGGBB для openpyxl).

Устаревшие ключи `fill_accessibility_ok` / `fill_accessibility_fail` используются только как запасные, если новые ключи не заданы.

## 9. Красный (пастельный) заголовок колонки — `fill_header_stock_out`

### 9.1. Когда красится строка 1

Для каждой **колонки ITEM** заголовок (ячейка row=1) получает заливку **`fill_header_stock_out`** (по умолчанию `E8B4B8`), если **хотя бы у одного менеджера** на листе RATING для этого кода товара:

| `item_amount_scope` | Условие |
|---------------------|---------|
| **`global`** (по умолчанию) | `sum(заказов по коду в ORDER) >= itemAmount` |
| **`per_manager`** | есть строка с `count >= itemAmount` |

### 9.2. Пример ITEM_01_2025-2_08 (itemAmount = 5)

| Ситуация | Шапка | Ячейка другого менеджера |
|----------|-------|---------------------------|
| Менеджер A исчерпил itemAmount | Красная | **До правки:** Y если рейтинг OK. **После:** N/не «доступен», если у B count < limit |
| Менеджер A исчерпил, B никогда не заказывал | Красная | B: **Y** только если B проходит все критерии и **его** count < itemAmount |
| Нет itemAmount в JSON | Обычная | Только 4 состояния матрицы |

**Роль и период** из колонок «Наименование Роли» / «Период» в коде **не участвуют** в расчёте itemAmount (в плане упоминались для среза; фактически считается **только табельный + код товара** по ORDER). Если нужен срез роль+период — это отдельная доработка.

### 9.3. Как читать шапку вместе с ячейками

- **Красная шапка** = «на этом листе **кто-то** уже набрал максимум заказов по этому SKU (itemAmount)».
- **Y/N/число в строке** = ситуация **конкретного** менеджера (рейтинг, LIST-REWARDS, группы, **его** count, **его** лимит).

## 10. Сводная блок-схема для одной ячейки (строка i, колонка товара T)

```text
count := ORDER_filtered[табельный_i, код_T]
ordered := count > 0
base := item_accessible_for_manager(rules, ранги, кристаллы, ORDER_codes, LIST-REWARDS, табельный)
blocked := код_T в группе с исчерпанным лимитом заказов
global_sum := sum ORDER по коду T (все табельные)
limit := itemAmount
склад_исчерпан := global_sum >= limit   (при scope=global)

accessible := base AND NOT blocked AND NOT (склад_исчерпан AND count=0)

значение := count | "Y" | "N"
fill := f(accessible, ordered)

если склад_исчерпан → заголовок колонки T ← fill_header_stock_out
```

## 12. Отладка расхождений «ожидание vs Excel»

| Симптом | Что проверить |
|---------|----------------|
| Везде N, рейтинг хороший | Пороги minRating*; пустые места в колонках; LIST-REWARDS / rewardCode |
| Y, но шапка красная | Другой менеджер исчерпил itemAmount; у вас count < limit — **норма** после правки |
| Y при 5 заказах от 5 людей, itemAmount=5 | Проверьте `item_amount_scope`: нужен **`global`**, не per_manager |
| Шапка красная, у вас count = 0 | Норма для global: склад пуст, у вас нет заказа → **N** |
| Число заказов, розовый фон | ordered + не accessible (группа, рейтинг или лимит после заказа) |
| Колонки ITEM нет | REWARD_TYPE не ITEM; нет разворота ADD_DATA |
| Неверный count | Дубликаты табельного в ORDER; код товара не совпадает с REWARD_CODE; статус не отфильтрован |

Логи: `[rating_item_matrix]`, `[reward_item_catalog]` уровень INFO/WARNING/DEBUG.
