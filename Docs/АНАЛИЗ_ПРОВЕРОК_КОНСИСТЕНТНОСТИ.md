# Анализ проверок на консистентность

Документ содержит: описание **текущих** проверок консистентности (config.json → consistency_checks.rules), анализ связей между листами и **предложения новых проверок** для максимального охвата целостности данных.

Дата анализа: 2026-01-31.

---

## 1. Типы правил (реализованы в коде)

| Тип | Назначение | Параметры правила |
|-----|------------|-------------------|
| **referential** | Внешний ключ в одну колонку: значения на листе A должны быть в справочнике (лист B) | `sheet_src`, `column_src`, `sheet_ref`, `column_ref` |
| **referential_composite** | Внешний ключ из нескольких колонок: комбинация колонок на A должна встречаться на B | `sheet_src`, `columns_src`, `sheet_ref`, `columns_ref` |
| **unique** | Уникальность комбинации колонок на листе (отсутствие дублей по ключу) | `sheet`, `key_columns` |
| **field_length** | Длина значений в полях не превышает/равна лимиту (оператор + лимит по каждому полю) | `sheet`, `result_column`, `fields` (поле → limit, operator) |

Общие поля: `id`, `name`, `enabled`, `output` (column_on_sheet, include_in_summary).

## 4. Связи между листами (по merge и ключам)

На основе `merge_fields_advanced` и `summary_key_defs`:

- **REPORT** формируется по ключам: CONTEST_CODE, TOURNAMENT_CODE, MANAGER_PERSON_NUMBER (и др.). Данные подтягиваются из CONTEST-DATA, TOURNAMENT-SCHEDULE, LIST-TOURNAMENT, EMPLOYEE (табельный → MANAGER_PERSON_NUMBER), LIST-REWARDS, REWARD-LINK, ORG_UNIT_V20 и т.д.
- **TOURNAMENT-SCHEDULE**: пара (TOURNAMENT_CODE, CONTEST_CODE) — естественный ключ; CONTEST_CODE должен быть в CONTEST-DATA (уже проверяется).
- **REWARD-LINK**: связи конкурс–группа–награда; все referential и unique уже заданы.
- **GROUP**, **INDICATOR**: привязаны к CONTEST-DATA по CONTEST_CODE.
- **EMPLOYEE**: PERSON_NUMBER, PERSON_NUMBER_ADD — справочник сотрудников; REPORT ссылается на него через MANAGER_PERSON_NUMBER.

## 5. Предлагаемые новые проверки консистентности

Цель: максимум проверок в рамках уже реализованных типов (referential, referential_composite, unique, field_length).

### 5.1. Referential (новые)

| id (предлаг.) | name | sheet_src | column_src | sheet_ref | column_ref | Обоснование |
|---------------|------|-----------|------------|-----------|------------|-------------|
| ref_report_contest | Отчёт: код конкурса в справочнике конкурсов | REPORT | CONTEST_CODE | CONTEST-DATA | CONTEST_CODE | REPORT подтягивает данные по CONTEST_CODE из CONTEST-DATA |
| ref_report_tournament | Отчёт: код турнира есть в расписании | REPORT | TOURNAMENT_CODE | TOURNAMENT-SCHEDULE | TOURNAMENT_CODE | Каждый турнир в отчёте должен быть в TOURNAMENT-SCHEDULE |
| ref_report_manager | Отчёт: табельный руководителя есть в справочнике сотрудников | REPORT | MANAGER_PERSON_NUMBER | EMPLOYEE | PERSON_NUMBER | MANAGER_PERSON_NUMBER приходит из EMPLOYEE (PERSON_NUMBER); при наличии колонки PERSON_NUMBER_ADD можно добавить отдельную проверку или проверять по объединению двух колонок* |
| ref_group_indicator | Индикатор: код конкурса из индикатора есть в группах (у конкурса есть группы) | INDICATOR | CONTEST_CODE | GROUP | CONTEST_CODE | Смысловая целостность: у конкурса с индикаторами должны быть группы |
| ref_schedule_reward | Расписание: код конкурса из расписания есть в связях наград (опционально) | TOURNAMENT-SCHEDULE | CONTEST_CODE | REWARD-LINK | CONTEST_CODE | Слабая связь: не каждый конкурс обязан быть в REWARD-LINK; можно включить как предупреждающую проверку «есть ли хотя бы одна связь наград»** |

\* Проверка ref_report_manager: если в EMPLOYEE есть и PERSON_NUMBER, и PERSON_NUMBER_ADD, то «справочник» допустимых табельных — объединение двух колонок. Текущий тип referential проверяет одну колонку справочника; для двух колонок потребуется либо расширение типа (referential_multi_column_ref), либо два правила (по PERSON_NUMBER и по PERSON_NUMBER_ADD) с дублированием нарушений. Рекомендация: одно правило REF: REPORT.MANAGER_PERSON_NUMBER → EMPLOYEE.PERSON_NUMBER (основной табельный).

\** Или не добавлять, если бизнес-логика не требует.

### 5.2. Referential_composite (новые)

| id (предлаг.) | name | sheet_src | columns_src | sheet_ref | columns_ref | Обоснование |
|---------------|------|-----------|-------------|-----------|-------------|-------------|
| ref_comp_report_schedule | Отчёт: пара турнир+конкурс есть в расписании | REPORT | TOURNAMENT_CODE, CONTEST_CODE | TOURNAMENT-SCHEDULE | TOURNAMENT_CODE, CONTEST_CODE | Строка отчёта должна соответствовать существующей паре турнир–конкурс в TOURNAMENT-SCHEDULE |
| ref_comp_schedule_group | Расписание: у пары турнир+конкурс есть группы (опционально) | TOURNAMENT-SCHEDULE | CONTEST_CODE, TOURNAMENT_CODE | GROUP | CONTEST_CODE, … | В GROUP нет TOURNAMENT_CODE — только CONTEST_CODE, GROUP_CODE, GROUP_VALUE. Поэтому составной ключ TOURNAMENT+CONTEST в GROUP не проверить без нового поля. Не добавляем. |
| ref_comp_indicator_group | Индикатор: пара конкурс+тип расчёта есть в индикаторах (уникальность уже есть; это обратная проверка «все пары из GROUP…» — не применимо). | — | — | — | — | Не требуется. |

Итого новых referential_composite: **1** (ref_comp_report_schedule).

### 5.3. Unique (новые)

| id (предлаг.) | name | sheet | key_columns | Обоснование |
|---------------|------|-------|-------------|-------------|
| unique_group_contest | Группа: нет дублей по конкурс+группа (без значения) | GROUP | CONTEST_CODE, GROUP_CODE | Жёстче, чем 3: одна пара конкурс+группа не должна повторяться с разными GROUP_VALUE (если бизнес это запрещает). Иначе дубли по (CONTEST_CODE, GROUP_CODE) с разными GROUP_VALUE уже ловит правило 3. |
| unique_indicator_contest_n | Индикатор: нет дублей по конкурс+N (одна позиция N на конкурс) | INDICATOR | CONTEST_CODE, N | Уже есть unique_indicator_n по N и unique_indicator_1 по конкурс+тип+код. По бизнесу может требоваться уникальность (CONTEST_CODE, N). |

При необходимости можно добавить unique по другим листам, если появятся ключевые поля (например, LIST-TOURNAMENT, LIST-REWARDS — после уточнения первичных ключей).

Итого новых unique: **0–2** (по желанию).

### 5.4. Field_length (новые)

| id (предлаг.) | name | sheet | result_column | fields (поле → limit, operator) | Обоснование |
|---------------|------|-------|---------------|----------------------------------|-------------|
| field_length_contest | Конкурсы: длина кода/названия в норме | CONTEST-DATA | FIELD_LENGTH_CHECK | CONTEST_CODE ≤50, FULL_NAME ≤500 (лимиты подставить по регламенту) | Ограничение длины служебных и текстовых полей |
| field_length_group | Группа: длина полей в норме | GROUP | FIELD_LENGTH_CHECK | GROUP_CODE ≤20, GROUP_VALUE ≤200 | Аналогично |
| field_length_indicator | Индикатор: длина полей в норме | INDICATOR | FIELD_LENGTH_CHECK | INDICATOR_CODE ≤50, N ≤10 (как строка) | По регламенту |
| field_length_reward | Награды: длина кода/названия в норме | REWARD | FIELD_LENGTH_CHECK | REWARD_CODE ≤50 | По регламенту |
| field_length_reward_link | Связь наград: длина кодов в норме | REWARD-LINK | FIELD_LENGTH_CHECK | CONTEST_CODE ≤50, GROUP_CODE ≤20, REWARD_CODE ≤50 | По регламенту |
| field_length_schedule | Расписание: длина кодов в норме | TOURNAMENT-SCHEDULE | FIELD_LENGTH_CHECK | TOURNAMENT_CODE ≤50, CONTEST_CODE ≤50 | По регламенту |
| field_length_org_unit_ext | Орг.единицы: доп. поля (если есть иные длинные поля) | ORG_UNIT_V20 | — | По текущему конфигу уже 3 поля; при появлении новых — добавить в fields |
| field_length_user_role | Роли: длина номера правила | USER_ROLE | FIELD_LENGTH_CHECK | RULE_NUM ≤20 | По регламенту |
| field_length_user_role_sb | Роли СБ: длина номера правила | USER_ROLE SB | FIELD_LENGTH_CHECK | RULE_NUM ≤20 | По регламенту |

Лимиты (50, 20, 200, 500 и т.д.) следует уточнить по реальным регламентам и длинам полей в БД/интерфейсе.

Итого новых field_length: **до 9** (по одному правилу на лист с ключевыми полями).

## 7. Рекомендуемый порядок внедрения

1. **Обязательно (высокий эффект):**
   - ref_report_contest (REPORT → CONTEST-DATA)
   - ref_report_tournament (REPORT → TOURNAMENT-SCHEDULE)
   - ref_comp_report_schedule (REPORT: пара TOURNAMENT_CODE+CONTEST_CODE в TOURNAMENT-SCHEDULE)
   - ref_report_manager (REPORT.MANAGER_PERSON_NUMBER → EMPLOYEE.PERSON_NUMBER)

2. **Желательно:**
   - ref_group_indicator (INDICATOR.CONTEST_CODE есть в GROUP)
   - field_length по листам CONTEST-DATA, GROUP, INDICATOR, REWARD, REWARD-LINK, TOURNAMENT-SCHEDULE, USER_ROLE, USER_ROLE SB (лимиты уточнить).

3. **По необходимости:**
   - unique_group_contest, unique_indicator_contest_n;
   - ref_schedule_reward.

## 9. Итог

- **Текущих проверок: 27** (7 referential, 1 referential_composite, 16 unique, 3 field_length).
- **Предложено новых: от 5 до 17** в зависимости от включения опциональных и field_length по всем листам.
- После внедрения рекомендуемого набора (как минимум 4 referential + 1 referential_composite) общее число проверок может составить **32+**, с расширением field_length и при необходимости unique — **40+**.

Все новые правила используют только существующие типы правил в `consistency_checks`; доработка кода не требуется, достаточно дополнения `config.json` и при необходимости уточнения лимитов длины полей по регламенту.
